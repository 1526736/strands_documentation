Y1 Security Scenario - changes to the system and summaries of testing days

Thursday 12/6/14

  • Robot undocking and ran successfully until 3pm or so. While trying to diagnose lack of USB sound, killed pulse audio which screwed up USB bus and needed bob reboot.

Wednesday 11/6/14

  • 09:45 Arrived to find robot PC had been turned off because it was shaking outside the office at about 09:15. Restarted complete system. Likely fault: undocking did not stop, kept publishing cmd_vel and interfering with move_base.
  • 10:35 Robot back in action

Tuesday 10/6/14

  • Complete expert free day, Bruno available by mobile but not needed.

Monday 9/6/14

  • 17h45: Restarting the routine sent new tasks to the scheduler, and at 17h45 the robot was sent to the station. This created problems with the MDP, which in turn made the visual docking stop working (preemption problem I think, similar as the problems the door pass was having). Restarted the bringup and docked by adding a wait action using example_add_client.py.
  • 17h15: Edited routine to start at 9h, work on 10/06/14 and don't execute afternoon_slump_2 tasks, so we are sure that someone will be here to help him if he has troubles in the final day tasks.
  • 15:50: New python version of door pass tested and debugged. This is the version now running in the robot. on bob t-mux tab 7 rosrun scitos_door_pass door_pass.py

Friday 6/6/14

  • 08:30: Some night task could not be preempted. No intervention made.
  • 08:45: Day started as planned, FE check action server failed, fixed bug and restarted that server
  • 10:02: Morning task done, Bob docked.
  • 10:22: Robot eye lid fixed while charging by removing head and repositioning stopper.
  • 16:09: Schedule changed to take 10/06/14 as day off, and start at 9.30am
  • 16:30: Adding logging to instance checking, temporarily missing action server cause executive to crash; restarted executive.

Thursday 05.6

  • 14h46: Door pass node broken, pushed robot through door and restarted node.
  • 14h00: Killed door pass for debugging
  • 09h30: Fixed recognition node by removing /home/strands/sift_flann.idx and /home/strands/object_instance_models/sift_trained/telephone.pcd/sift_flann.idx. Relaunched new version of checkObjectPresence node
  • 08h45: Absence of checkObjectPresence action crashed executor node. Restarted it with the old version.

Wednesday 04.6

  • 15h30: Restarted fire extinguisher action server, with up-to-date action server script. Old version was being run. Recognition node still crashing.
  • 14h40: Fire extinguisher detection failed - as it has been doing since the nodes in bobl had to be restarted on 03.6 - but this time it got stuck preempting. The robot stood there for 15 min. so i had to restart the checkObjectPresence node. The robot continued afterwards, but the node died (seg fault) on another check, and had to be re-run again.
  • 12h45: Restarted routine. Tasks from 14h30 were scheduled.
  • 11h00: Stopped routine for BBC filming

Tuesday 03.6

  • 16h00: Stopped routine to take pictures next to G4S logos. Losing network also made tmux in bobl crash. re-launched all nodes.
  • 12h10: Re-started cloud_merge so it saves intermediary data, as requested by Rares
  • 09h50: Recalibrated chest camera
  • 09h30: Door passing had to be restarted again

Monday 02.6

  • 16h15: Attempted to debug door passing but was not able to recreate the bug. Restarted navigation to get Waypoints 17 and 18 updated.
  • 14h50: Another global plan failure, which make the robot cancel some of its tasks. The positions of WayPoints 17 and 18 were slightly changed to try and avoid this in the future.
  • 14h15: Door passing stopped working again and had to be restarted. The queue sizes for /scan and /robot_pose were changed to 1.
  • 09h40: Global plan fail problem again, had to push the robot without it asking for help.

Friday 30.5.

  • 17h30: Intervention: Changed default time before scheduling to 15 min before start window of tasks; made door passing fail when getting further away from goal instead of when it's further away than a given threshold. Restarted door_passing, scheduler/execution and routine nodes.
  • 15h00. Problem: Scheduling during execution of other tasks made the socket communication with the prism server lose sync, which then caused the executor node to crash. Expert Solution: Had to kill all the scheduler nodes. For next week, we should make sure that scheduling only happens when the robot is not navigating. Also restarted routine to try to have tasks of the 14h30 slot done, but it was unfeasible.
  • 12h10. Problem: Messed up waypoint visualization while playing with rviz. Expert Solution: Re-ran visualizer node.
  • 10h00. Problem: Root partition in the robot was getting full from tmp data dumped by the replicator node. Expert Solution: Changed tmp directory to /data and edited ros_datacentre/replicator_node.py accordingly. Restarted the replicator node.

Thursday 29.5.

  • 09h15. Problem: Global planner failure because a door was over the waypoint. Robot just got there failing navigation actions without asking for help, which lead to some of the perceive_tabletop actions being cancelled. Expert Solution: Opened the door.
  • 11h40. Problem: Bug on nav help when asking for help more than 5 times. Most of the tasks at this time slot were not performed Expert Solution: Had to kill the nav nodes, debug and restart.
  • 12h10. Problem: New global planner failure, this time related to not clearing of costmap (not 100% sure). Once again no help requested. Expert Solution: Force pushed the robot until he managed to start moving again.
  • 12h40. Problem: Bug on nav help wasn't well solved. This only became apparent when the robot was coming to dock, so the tasks had all been performed. Expert Solution: Kill the nav nodes, debug and restart. Sent the robot to charging station using topological navigation action client.
  • 17h00. Problem: Known issue where robot was going back to normal navigation while being helped, and then not asking for help properly next time he needed to. Expert Solution: Debug after the day's tasks were finished. Edited monitored navigation to disallow preemption when the robot is being helped. Also ensured that all goals of the current continuous navigation action are cancelled when there is a new action to be executed. Stress tested the changes. The robot now recovers properly, and switches correctly to the continuous navigation action associated to the node he ends up after being helped.

Wednesday 28.5.

  • update parameters for tabletop, relaunched coverage_avg = 2.0 and coverage_total = 0.6 and roslaunch perceive_tabletop_action perceive_tabletop_params.launch
  • 11:25 We lost Wifi connection on robot, then tmux on bob crashed (maybe related?). Restarted all the system on Bob and Bobl
  • 11:50 The schedule for 11:30 is meanwhile infeasible.
  • edited routine script to avoid waiting task in charging station are not resent when the robot is already there
  • fixed concurrency outcomes in executor state machine
  • relaunched both (routine and task executor)
  • got WP 37 and 38 closer together to avoid failure in door passing between them

Tuesday 27.5.

Changes:

  • 11:00 Restarted table top - new version included
  • 12:15 Created On demanded tabletop task and fire extinguishers
  • 13:45 Updated MDP to correct expected travel times, restarted node
  • 14:19 Updated task_executor to sm_executor branch in Nick's fork, restarted executor and routine
  • 15:00 We reverted from sm_executor branch and run routine script again
  • 15:53 Back to sm_executor, recompiled, rerun

Known issues:

  • Door passing stopped working, after restart of navigation worked; the robot was passing doors by "carpet recovery behaviour"
  • The robot asks for a help but then starts to move (not waits for the actual help). Sometimes it also skips asking for helping and immediately shows the screen with "push me to free space", the wheels seems to be really blocked at that time. Maybe related to issue with not waiting for a help
  • Sometimes he stops at a final WP, which is not correct. This happens when the final WP is closed to another WP (robot stops there). Bruno has a clue.

TODO:

  • test sm_executor

Summary:

  • In the morning, Bob was executing tasks, having serious issues to pass doors. Tabletop observing was not working because of some bug.
  • Later, tabletop was not producing an error as we upgraded. However, it could not find a viewpoints, even * After lunch, day routine was not working at all. It was probably messed by on demand tasks, what we created.
  • Door passing worked properly after restart of navigation.

Friday 23.5.

Summary

  • The navigation is pretty reliable, can now leave it to do its routine. It gets stuck, but asks for help properly with sound and screen. Scheduling makes sense, can preemept the needed things. Fire extinguisher checking works but always fails to find the object – no shock.
  • It is set up for 8.45 → 17.45, each table 3 times a day, each hall way metric point 3 times a day, each office metric 2 times per day, door checks 3 times and fire extinguishers 3 times.
  • I started at 4pm today, forced to do stuff by extending day end then placing back to normal. It did ~1 ½ hours of tasks. Some doors were shut so they failed to get the stuff, some desks were went to but observation failed to find view cones. Metric mapping was ok.
  • A note about the PTU: if anything ever drives it past the limit it screws everything up until a full restart of the scitos launch (PTU). So if you add tasks be sure they don't mess about.

To fix:

  • table observation index error
  • check mdp long traversal is blogging time right, also table observations same question.
  • Possibly improve instance check as it runs but never finds the thing and blog is messy
  • any further strange mdp bugs

Thursday 22.5.

The robot is docked for the night, due to restart tomorrow at 8.45 (can't get into building until 8.30). The day got off badly, but all in all the navigation is ok now - at least there are less bugs than this morning.

Key points:

  • the USB sound has to be unplugged when booting; USB setup is a mess and need looking into.
  • the schedule must be 8.45 start not 8.30 as we cant get in until 8.30
  • On Friday, Chris will check that he continues his schedule at 8.45, and that night tasks were ok. Chris will test the topological same-node fix and the monitored navigation logging update. Then Chris will start a complete schedule and just try and help it out when it asks for help. This should be by midday.
